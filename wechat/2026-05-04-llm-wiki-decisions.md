# 我用 Github + Claude Code 搭了一个会持续进化的 AI 知识库，实战级教程来了

最近看到一篇文章，讲怎么用 Obsidian + Codex 搭一个会进化的 AI 知识库。

看完我没急着照做。

因为我意识到一件反直觉的事：

**我的 GitHub 文档库其实已经是知识库了。缺的不是工具，是工作流。**

---

## 我的起点：不是空仓库

先说我的起点。

不是空仓库。也不是新开 Obsidian。

是一个我已经维护了几个月的 GitHub 文档库，长这样：

```
├── setup/        ← 安装与初始配置
├── usage/        ← 命令与快捷键
├── channels/     ← 远程控制
├── plugins/      ← 插件指南
├── tools/        ← 具体工具
├── quickstart/   ← 快速入门
├── workflow/     ← 工作流与方法论
├── internals/    ← 内部原理
└── CLAUDE.md     ← 给 Claude 的工作规则
```

结构有了。

分类有了。

连给 AI 的工作规则都有了（`CLAUDE.md`，作用相当于原文里的 `AGENTS.md`）。

但每次我想往里加东西，都还是要手动判断：

> "这篇文章该放 setup 还是 workflow？"
> "要不要更新 README 索引？"
> "这个概念跟现有哪篇文档相关？"

**每一次都重新想一遍。**

这就是问题所在。

不是我的目录结构不够好。

是我"消化新资料"的判断标准，从来没有被固化下来。

---

## 真正缺的，是 ingest 流程

看完那篇 LLM Wiki 文章我才反应过来。

我缺的不是分类目录。

是一套**资料消化流程**。

于是我打开 Claude Code，让它帮我起草一份 ingest 规则。

它没有马上动笔。

它反过来问了我 5 个问题。

每个问题，原文都没说。

但你不答，规则就跑不起来。

---

## 决策 1：新文章进来，合并还是新建？

第一个问题：

> 一篇新原文 ingest 进来，是合并到老文档里，还是新建一篇？

三个选项：

| 选项 | 描述 | 代价 |
|------|------|------|
| A) 严格阈值 | 重叠 ≥70% 才合并 | 容易碎片化 |
| B) 宽松阈值 | 主题相关就合并，新建需明确理由 | 单文档会变长 |
| C) 主题导向 | 同分类下有相邻主题就合并 | 分类粒度依赖人 |

**我选 B。**

为什么？

我那 8 个分类目录已经划好了"主题边界"。

同一分类下再分文档，多半是过度细分。

我宁可一个文档变长。

也不想以后有 5 篇只差一点的近义文档。

---

## 决策 2：跨分类的概念怎么放？

下一个问题：

> 一个概念跨多个分类（比如"LLM Wiki 方法论"既属于 workflow 又涉及 tools），怎么办？

| 选项 | 描述 |
|------|------|
| A) 单一归属 + 交叉链接 | 只放主分类，其他分类用 markdown 链接指过来 |
| B) 多处放置 | 每个相关分类都放一份 |
| C) 主分类 + README 多入口 | 文档放主分类，README 在多个章节都列出来 |

**我选 A。**

B 是噩梦。

同步两份文档的成本，会让你彻底放弃维护。

C 也行。但 README 索引会越长越乱。

A 最干净。

每个概念**只有一个权威落点**，别处只留指针。

---

## 决策 3：原文链接保留吗？

| 选项 | 描述 |
|------|------|
| A) 全部保留外链 | 可追溯，但公众号链接易失效 |
| B) 保留链接 + 在 raw/ 存原文备份 | 最可靠，仓库会膨胀 |
| C) 只保留标题和作者，不存链接 | 精简，失去溯源 |

**我选 C。**

想一下：

公众号文章下架是常态。

我现在保留链接，未来 90% 都会变死链。

死链让读者点了空欢喜。

还不如不留。

原文如果真有价值，会通过 git 历史保留下来 —— 这就引出下面两个决策。

---

## 决策 4：ingest 完成后，raw 文件怎么处理？

| 选项 | 描述 |
|------|------|
| A) 原地保留 | 可反复 ingest，raw/ 越来越大 |
| B) 移到 raw/processed/ | 标记已处理 |
| C) 删除 | 仓库干净 |
| D) 加注释 `<!-- ingested: ... -->` 后保留 | 留痕迹 |

**我选 C。**

A、B、D 看似稳妥。

实际是把"决策"推迟到未来。

raw 越积越多，最后还是得做清理决策。

C 一步到位：编译完就删。

但 C 有个前提条件。

待会儿讲。

---

## 决策 5：raw/ 进 git 吗？

| 选项 | 描述 |
|------|------|
| A) 进 git | 完整可追溯，公开仓库有版权风险 |
| B) 不进 git | 干净，失去原始素材的版本历史 |
| C) raw/ 进，raw/private/ 不进 | 分公开和私有 |

**我选 A。**

我的仓库虽然公开，但 raw 里的内容会很快被删除（决策 4）。

**留在工作树的时间窗口很短。**

版权风险被时间窗口压到几乎为零。

换来的好处：git 历史里完整保留了所有原文。

---

## 等等，5 个决策合起来藏了一个坑

5 个决策都拍完，准备动手时我才意识到一件事。

**决策 4（删除）和决策 5（进 git）合起来，藏着一个隐藏约束。**

看出来了吗？

如果我的流程是：

1. 把原文存进 `raw/`
2. 立刻 ingest
3. 删除原文

这个文件**从未进入 git 历史**。

删了就是真的没了。

所谓"git 历史溯源"瞬间作废。

正确的流程必须是：

```
1. 把原文存进 raw/
2. git commit raw/xxx.md     ← 这步不能跳
3. 让 Claude 跑 ingest
4. ingest 内部会删除 raw 文件
5. git commit 收尾
```

看到关键差别没？

**必须先 commit，再 ingest。**

我把这个约束**写死进了 ingest 规则的"前置检查"**：

每次 ingest 之前，Claude 都会先查 git，没 commit 就停下来叫我先 commit。

这种"两个看似无关的决策叠加产生的隐藏约束"，原文不会告诉你，AI 也不会主动提醒。

**只有真动手才会撞上。**

---

## 5 个决策定下来之后，神奇的事情发生了

我做了这些事：

1. 把那篇 LLM Wiki 文章存进 `raw/2026-05-04-llm-wiki-obsidian.md`
2. `git commit`（前置约束要求）
3. 让 Claude 调 `ingest-raw` skill

然后我什么都没干。

看着 Claude 自己：

- 扫描了我现有 8 个分类目录
- 判定 workflow/ 里只有"编程方法论"（vibe-coding-guide），没有"知识管理方法论"容器
- **新建** `workflow/llm-wiki-methodology.md`
- 在 `tools/obsidian-setup.md` 里加了一行交叉链接（决策 2 在生效）
- 自动更新 `README.md` 索引
- 删除 raw 原文（决策 4）

我只在最后做了一次 `git commit`。

**整个过程我没碰一次"该放哪、要不要加链接、什么时候删原文"的判断。**

全部被 5 个决策预先固化了。

---

## 这才是 LLM Wiki 真正强的地方

5 个决策回头看，你会发现一个共同点。

它们**都不是"知识管理"问题**。

全都是**工作流偏好问题**：

- 决策 1 = 我容忍单文档变长还是文档变多？
- 决策 2 = 我用链接换冗余，还是反过来？
- 决策 3 = 我接受死链率，还是接受信息精简损失？
- 决策 4 = 我把判断推到未来，还是现在解决？
- 决策 5 = 我用 git 当原始素材的存档，还是另存？

一旦把这些偏好写进 skill，未来每次 ingest 都不用再想。

Agent 替你执行。

你的工作量从"每次都判断一遍"变成"判断一次，固化下来"。

这才是 LLM Wiki 比传统笔记库强的地方：

**它不是多了个工具。是把你的判断标准，固化下来了。**

---

## 想抄作业？按这个顺序走

**一、别新建实验仓。**

如果你的笔记 / 文档已经有分类（哪怕乱也算有），直接在原仓库加 `raw/`。

原文那个最小实验仓是给完全没起步的人的。

已经起步的人继续踩，反而是浪费。

**二、5 个决策自己拍一遍。**

别照抄我的。

每个人工作流偏好不一样。

我选 B，你可能选 C。

重点不是抄答案。

是先想过这些问题。

**三、写一份 ingest 规则**（skill 也好，自定义命令也好，Cursor rule 也好）。

工具不重要。

但规则里**必须包含前置检查**。

否则会踩"决策 4 + 5 隐藏约束"的坑。

**四、挑一篇原文跑一次完整闭环。**

第一次完成度，比规模重要 100 倍。

**五、跑完之后回头改规则。**

第一次的规则一定不完美。

但它会教你下次该改什么。

---

我用了大概 1 小时把这套跑通。

比我原来"看一篇好文章就存到 Obsidian / Notion"的做法多花了 30 分钟。

但**未来每一次都比以前快 10 倍**。

知识库不是为了存得多。

是为了下次想用的时候，不用再翻。
